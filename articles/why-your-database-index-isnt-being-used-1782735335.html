Why Your Database Index Isn't Being Used

You added an index, the query is still slow, and the database seems to be ignoring all your careful work. This is one of the most common sources of confusion in backend development, and the reasons are surprisingly knowable.

The optimizer gets a vote

An index is an offer, not a command. The query planner weighs the cost of using it against the cost of a full scan, and on a small or low-variety table a sequential scan is often genuinely faster. If your test table has a thousand rows, don't be surprised when the index sits unused — there simply isn't enough data for it to pay off.

Functions on the column kill the index

The classic mistake: WHERE lower(email) = '...' or WHERE date(created_at) = '...'. The moment you wrap an indexed column in a function, the plain index on that column no longer matches, and the planner falls back to scanning. The fix is either an expression index on lower(email), or rewriting the query to compare the raw column against a range.

Leading-column rules for composite indexes

A composite index on (a, b) can serve queries filtering on a, or a AND b, but not b alone — the index is sorted by a first. People build a multi-column index and then filter only on the second column, then wonder why it's skipped. Order the columns by how you actually query them, most-selective and most-frequently-filtered first.

Type mismatches and stale statistics

Comparing a bigint column to a string literal can force an implicit cast that defeats the index. And if the table's statistics are stale, the planner may badly misjudge how many rows a condition will match. Run EXPLAIN ANALYZE, read what the planner actually decided, and refresh statistics if its row estimates are wildly off.

The habit that fixes all of this is the same: stop guessing, run EXPLAIN, and let the database tell you why it chose what it chose. The answer is almost always in the plan.
